3 Data Flow

1
User sets filter
User changes a dropdown in FilterBar. The composable's singleton ref updates immediately — all views that call useFilters() share the same reactive state.
useFilters() → selectedWarehouse.value = 'WH-A'
2
View re-fetches
Each view watches the filter refs. When they change, the view calls its load function which reads the current filters and builds query params.
watch([selectedWarehouse, selectedCategory], loadData)
3
API call via api.js
api.js constructs URLSearchParams from the filter object, skipping any filter set to 'all'. Axios sends the GET request to the FastAPI backend.
GET /api/inventory?warehouse=WH-A&category=electronics
4
Backend filters in-memory data
FastAPI receives query params, calls apply_filters() and filter_by_month() against the in-memory Python lists. No database query — pure list comprehensions.
apply_filters(inventory_items, warehouse='WH-A', category='electronics')
5
Pydantic validates & serializes
FastAPI validates the filtered list against the Pydantic response_model, serializes to JSON, and returns it. Invalid fields are stripped; missing required fields raise a 422.
response_model=List[InventoryItem] → 200 OK + JSON
6
Vue updates computed & template
The response is stored in a raw ref (e.g. inventoryItems). Computed properties derive filtered/sorted/aggregated views from the raw data. Templates reactively re-render.
inventoryItems.value = data → computed re-evaluates → DOM updates

5 Key Patterns

Singleton Filter Composable

useFilters() holds refs at module scope — shared state across all views without Vuex or Pinia. One source of truth for all 4 filters.

  • selectedPeriod, selectedLocation, selectedCategory, selectedStatus
  • getCurrentFilters() → ready-to-use object for api.js calls
  • hasActiveFilters computed → drives reset button visibility

Raw Ref + Computed Pattern

Views store raw API response in a ref. All derived data (filtered, sorted, aggregated) lives in computed properties — never in methods.

  • allOrders = ref([]) ← raw from API
  • filteredOrders = computed(() → derived)
  • Computed is cached; re-evaluates only when deps change

Server-Side Filtering

Filters are applied in Python on the server, not in the browser. Each filter change triggers a new API request with updated query params.

  • apply_filters() chains warehouse → category → status
  • filter_by_month() handles YYYY-MM and Q1-2025 quarter strings
  • inventory endpoint does not support month filter (no time dimension)

Tasks: Dual Source

Tasks are merged from two sources in App.vue: mock tasks from useAuth() (local) and persisted tasks from the API. Merged via computed.

  • currentUser.value.tasks — in-memory mock, mutated directly
  • apiTasks.value — fetched from /api/tasks on mount
  • tasks = computed(() → [...mockTasks, ...apiTasks])
